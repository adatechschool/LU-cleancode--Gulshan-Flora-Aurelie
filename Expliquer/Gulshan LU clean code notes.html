Clean Code

Clean code is a reader-focused development style that produces software that's easy to write, read and maintain. Must be expressive. Must be able to read it the first timr you see the code and know what it is about. one class or function should have 1 responsibility. Don't repeat: create a function to repeat instead of copying code.

  • Break down big code into smaller sections
  • Time limit and dont think about it
  • YAGNI - code onky necessary parts, dont anticipate
  • software design patters
  • If you want to learn something, practice. Make examples and make them work because read about something is not enough.
  • When developing a web application, think mobile-first and the associated power and bandwidth restrictions.

SOLID

  1. S - Single-responsiblity Principle
  2. O - Open-closed Principle
  3. L - Liskov Substitution Principle
  4. I - Interface Segregation Principle
  5. D - Dependency Inversion Principle

Used to fid design problems
With these principles it is easier to understand and change code. Makes code more flexible.
Need a context to know when and where you need refactoring

Vocab: code spaghetti, dirty code, smelly code

Technical Debt : the process of prioritising speedy delivery over perfect code.

If you choose to have technical debt, there should be strategy, intention and rationale behind it.
Is it wreckless VS prudent?
Is it inadvertent or deliberate?

Technical debt should be context-based, to see whether it is necessary and should be implemented or not
